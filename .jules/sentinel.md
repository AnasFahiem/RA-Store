## 2024-05-24 - [CRITICAL] Missing Authorization Checks in Admin Server Actions Bypass RLS
**Vulnerability:** Mutation Next.js Server Actions (e.g., `addHeroSlide`, `deleteHeroSlide`, `addHeaderSlide` in `hero.ts` and `header.ts`) that utilized the service-role Supabase client (`createAdminClient()`) were missing manual role-based authentication/authorization checks. Because `createAdminClient()` bypasses database-level Row Level Security (RLS) entirely, these server actions could be invoked directly by unauthenticated users or users with standard privileges, allowing unauthorized modifications of administrative configuration data.
**Learning:** Using a service-role client within Next.js Server Actions constitutes a severe security risk unless strict, explicit session and role validations (e.g., `const session = await getSession(); if (session?.role !== 'admin' && session?.role !== 'owner') return { error: 'Unauthorized' };`) are manually enforced inside every action. RLS provides no defense when the service-role is used.
**Prevention:** Always enforce explicit authorization checks via `getSession()` at the beginning of any Next.js Server Action that modifies data, especially if using an administrative or service-role database client. Default to standard client (which respects RLS) whenever administrative override is not strictly necessary.
